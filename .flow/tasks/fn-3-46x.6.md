# fn-3-46x.6 Design MCP tools partial spec for RadFlow

## Description
Analyze whether RadFlow needs MCP tools or if Claude Code skills are sufficient.

### Key Question

RadFlow is a **context-provider** (clipboard-first, not direct-write). Given this:
- Are MCP tools necessary?
- Or are Claude Code skills/plugins sufficient?

### Analysis Framework

**1. First: Determine if MCP is needed**

Consider:
- RadFlow provides context, doesn't write code directly
- Running locally (not remote service)
- Primary workflow: Select → Copy → Paste into AI → AI edits files
- Skills can call local binaries, read files, etc.

**2. If skills are sufficient:**

Use Pencil's MCP as **reference for creating skills**:
- What capabilities would skills need?
- Map Pencil's 14 MCP tools to potential skill equivalents
- Focus on read-only operations (get tokens, get components, get context)

**3. If MCP is needed:**

Compare approaches:
- MCP: Cross-tool compatibility (Cursor, Windsurf, etc.)
- Skills: Claude Code specific, simpler implementation
- Hybrid: Some MCP for cross-tool, some skills for Claude Code

### Pencil MCP Tools Reference

From task 2, catalog which tools are relevant:
1. batch_get - Read nodes (relevant for context)
2. get_editor_state - Current selection (relevant)
3. get_variables - Token values (relevant)
4. get_screenshot - Render to image (maybe relevant)
5. etc.

### Output

- Recommendation: MCP, skills, or hybrid
- If skills: proposed skill set based on Pencil reference
- If MCP: proposed read-only tool set
- Scope: read-only (no write operations per interview)
