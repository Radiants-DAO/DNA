The Architecture of Portable Design Systems

Design tokens provide the foundational vocabulary for a design system. When properly structured, they enable themes to be portable across frameworks, platforms, and rendering contexts. The two-tier model separates brand primitives from semantic mappings, creating a clean interface between identity and implementation.

This separation matters because it allows the same component to render correctly in a newspaper layout, a desktop application, or a mobile screen without any changes to its structural code. The semantic layer absorbs the visual variation while the component layer stays stable.

Building Portable Design Systems with DNA

The Design Nexus Architecture provides a standardized approach to theme packages that work across frameworks. At its core is a two-tier token system that separates brand identity from semantic purpose.

Token Architecture

Every theme begins with brand primitives: the raw color values, type choices, and spatial constants that define a visual identity. In RDNA, these live as CSS custom properties inside a Tailwind v4 @theme block.

Tier 1: Brand Primitives

Brand tokens are the immutable palette. They carry names that describe what they are, not what they do: --color-sun-yellow, --color-ink, --color-cream.

Why OKLCH?

The Radiants theme uses OKLCH color values because they provide perceptual uniformity across hue rotations. A 10% lightness shift in OKLCH looks like a 10% lightness shift to human eyes, which isn't true of HSL or hex values.

Tier 2: Semantic Mappings

Semantic tokens describe purpose: --color-page for backgrounds, --color-main for primary text, --color-line for borders. Components reference only these semantic names.

The semantic layer is the contract between design and engineering. Change the brand tokens, and the entire system updates. Change a semantic mapping, and a single role shifts.

Typography in the System

The typography scale defines seven steps from text-xs (10px) to text-3xl (32px). Three font families serve distinct roles:

  • Mondwest handles body text, providing readability at paragraph length
  • Joystix Monospace serves headings and UI labels with a distinct pixel-art character
  • PixelCode renders code blocks and technical content in a monospaced grid
Weight Availability

Note that Joystix Monospace ships only at weight 400. The typography.css base layer applies font-bold (700), font-semibold (600), and font-medium (500) to headings, but the font file cannot render these weights natively. The browser synthesizes bold from the 400 weight, which produces unpredictable results with pixel fonts.

Related: ESLint Enforcement

The rdna/no-hardcoded-typography rule prevents arbitrary font sizes and weights. Only token-mapped classes are allowed: text-sm through text-3xl.
