DNA vs shadcn/ui v4

Strategic Comparison and Adoption Analysis

DNA (Design Nexus Architecture) is a theme-system specification for portable design systems. It defines a two-tier token architecture, a three-file component pattern, and multi-theme identity packages (@rdna/radiants, @rdna/monolith). Its strength is token depth -- motion, elevation, typography, z-index, and accessibility tokens that no other system ships.

shadcn/ui v4 is the dominant component library for the React/Tailwind ecosystem. Its v4 release brings a registry protocol, MCP server, Skills for AI agents, preset codes, and OKLCH colors. Its strength is distribution infrastructure -- a mature CLI, community registries, and seamless AI integration that DNA has only planned.

The strategic question: DNA has richer design tokens and a tighter spec. shadcn has the distribution machinery and AI integration. Rather than choosing one, the optimal path is a bridge strategy -- keep DNA's token depth and spec rigor, adopt shadcn's distribution infrastructure, and compile one into the other.

05 Strategic Recommendations

ADOPT from shadcn

Replace DNA equivalent -- shadcn is mature where DNA is planned.
  • CLI tooling shadcn CLI is mature with init/add/view/search/build/migrate. DNA CLI is unbuilt.
  • Registry protocol for component distribution 12 item types, auth, namespaces. DNA has no distribution mechanism.
  • Preset system for theme distribution One-command setup. DNA's dna init was planned but never implemented.
  • MCP server for AI agent integration Working shadcn mcp init. DNA has no MCP server.
  • Skills system Replaces planned ai-skills/ directory. Supports 5+ AI clients.
  • OKLCH color space Perceptually uniform. Better interpolation and palette generation than hex/rgba.
  • RTL support DNA has no RTL story. shadcn has first-class --rtl flag and migration tools.

KEEP from DNA

shadcn has no equivalent -- these are DNA's unique value.
  • Two-tier token architecture (brand → semantic) shadcn's flat tokens can't cascade. Changing one brand color in DNA updates all semantic uses.
  • Multi-theme identity system radiants/monolith as separate packages. shadcn has no multi-theme concept.
  • Motion token system 5 durations, 3 easings, reduced-motion, Remotion adapter. shadcn has nothing.
  • Elevation/shadow system Mood-carrying behavior (pixel offsets in sun, glows in moon). Unique to DNA.
  • Typography scale + Z-index + A11y tokens Three entire token categories that shadcn delegates to Tailwind defaults or ignores.
  • Sun/Moon mode personality distinction Dark mode as a different identity, not just inverted colors.
  • .dna.json token binding format Maps variants to tokens per component. Essential for AI-assisted theming.
  • CI-enforced token governance + Design spec theme-spec.md with MUST/SHOULD/MAY. Machine-verifiable conformance.

BRIDGE: DNA Spec, shadcn Format

Compile DNA's depth into shadcn's distribution layer.
  • DNA tokens → shadcn cssVars Compile DNA's two-tier brand→semantic tokens into shadcn's flat cssVars format. DNA remains the authoring format; shadcn is the output target.
  • DNA components → shadcn registry Publish @rdna components as registry items. Users install via npx shadcn add @rdna/button.
  • dna.config.json → preset code Generate shadcn preset codes from DNA config. One-command DNA theme setup.
  • .schema.json → Skills enhancement Feed DNA's richer structured schema data into shadcn's Skills context for more reliable AI generation.

BUILD: Missing from Both

Gaps that neither system addresses today.
  • Token validation in shadcn shadcn has no required minimums. Apply DNA's spec rules as a shadcn plugin or CI step.
  • Motion/elevation tokens in shadcn shadcn has no motion system. Build a bridge that outputs DNA's motion tokens as shadcn theme extensions.
  • Multi-theme runtime switching shadcn is init-time only. DNA has packages but no runtime switcher. Build a theme loader.
  • DNA CLI → shadcn plugin Instead of building a separate DNA CLI, extend shadcn's CLI with dna validate, dna catalog as subcommands.

06 Migration Path

A phased approach that preserves DNA's token depth while progressively adopting shadcn's infrastructure. Each step is independently shippable. Total estimated effort: 4--6 weeks.

1
Adopt OKLCH in DNA tokens
Convert all hex/rgba values in tokens.css and dark.css to OKLCH. Preserve the two-tier brand→semantic architecture -- only the color values change, not the structure. Use oklch() with fallback for older browsers. Test contrast ratios against WCAG 2.1 AA.
Low effort -- ~2 days
2
Create @rdna shadcn registry
Set up a shadcn-compatible registry endpoint that serves @rdna components. Each component needs a registry:ui entry with its files, dependencies, and cssVars. Host on GitHub Pages or Vercel. This unlocks npx shadcn add @rdna/button.
Medium effort -- ~1 week
3
Generate preset code from dna.config.json
Build a compiler that reads dna.config.json and outputs a shadcn preset code. This lets new projects bootstrap DNA themes with one command. Map DNA's brand tokens to shadcn's cssVars format. Include DNA's extra token categories (motion, elevation, z-index) as CSS custom properties outside shadcn's scope.
Medium effort -- ~3 days
4
Publish components as registry items
Convert the 25 radiants components to shadcn registry format. Each component gets a registry.json entry. Include .dna.json as a metadata file (shadcn supports arbitrary file types in registries). Domain-specific components (Web3ActionBar, WindowManager) go in a separate @rdna/domain registry.
High effort -- ~1.5 weeks
5
Port .schema.json into Skills context
Enhance the shadcn Skills integration with DNA's richer schema data. Instead of Skills parsing JSX to infer component APIs, feed .schema.json structured data (props, slots, examples) into the Skills context. This gives AI agents more reliable component knowledge.
Medium effort -- ~3 days
6
Deprecate planned DNA CLI in favor of shadcn CLI
Stop planning the standalone dna CLI. Instead, build DNA-specific functionality as shadcn CLI extensions or standalone validation scripts. dna validate becomes a CI step. dna catalog becomes a registry build script. dna init becomes a shadcn preset code.
Low effort -- ~1 day (decision + docs)
7
Keep token spec, dark.css, and .dna.json as DNA-unique value
These are DNA's moat. Continue investing in theme-spec.md, the Sun/Moon personality system, motion/elevation/z-index tokens, and the .dna.json token binding format. These layers sit above shadcn's infrastructure and differentiate DNA from every other component library.

07 Risk Assessment

High Risk
shadcn's registry:base evolves to cover DNA's gaps
With extends: "none", shadcn already lets you build an entirely new design system on its infrastructure. If they add motion tokens, elevation, and multi-theme support, DNA's unique value shrinks significantly.
Mitigation: Ship the bridge fast. Get @rdna components into the shadcn registry ecosystem before shadcn builds its own token depth. First-mover in "rich tokens on shadcn infra" is defensible.
High Risk
GPL-3.0 license conflict with MIT ecosystem
@rdna/radiants is GPL-3.0. shadcn/ui and its ecosystem are MIT-licensed. Publishing @rdna components to a shadcn registry may create license compatibility issues for consumers who expect MIT.
Mitigation: Consider dual-licensing @rdna components, or re-licensing the registry-distributed versions under MIT while keeping the spec and token system under GPL-3.0. Consult legal before publishing to public registry.
Medium Risk
Maintaining two systems adds complexity
Running DNA tokens + shadcn distribution means maintaining a compilation pipeline, keeping token formats in sync, and debugging issues that could originate in either layer. Team cognitive load increases.
Mitigation: Build the bridge as a one-directional compiler (DNA → shadcn). DNA is always the source of truth. Automate the compilation in CI so the shadcn output is never manually edited.
Medium Risk
Community may not adopt DNA's richer token vocabulary
shadcn's community is used to ~25 flat tokens. DNA's ~50+ semantic tokens with motion, elevation, and z-index may feel over-engineered to developers who just want a button. Adoption friction could limit the registry's reach.
Mitigation: Make the extra tokens progressive. Components work with shadcn's base tokens. DNA tokens are optional enhancements that activate richer behavior. Don't require DNA tokens for basic usage.
Medium Risk
shadcn v4 breaks DNA token mappings
shadcn v4 changed token names and introduced OKLCH. Future versions may change again. The DNA→shadcn bridge could break on major shadcn releases.
Mitigation: Pin to shadcn v4 format. Version the bridge compiler. Add integration tests that verify the compiled output works with a reference shadcn project.
Low Risk
OKLCH browser support gaps
OKLCH is supported in all modern browsers (Chrome 111+, Firefox 113+, Safari 15.4+). Legacy browser users may see broken colors.
Mitigation: Use @supports with hex fallbacks, or rely on PostCSS postcss-oklab-function for automatic fallback generation. Acceptable for DNA's target audience.
